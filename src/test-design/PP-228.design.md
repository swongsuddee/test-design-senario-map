**Platform:** Web (Backoffice — Organizer Portal)
**Framework:** WebdriverIO + Appium + Mocha (TypeScript)
**App:** POPPA Backoffice (Organizer)
**Scope:** Resend verification email flow — countdown timer, button state, token invalidation, rate limiting, token expiry, and feedback toasts

---

## A. Technique Selection

| Module | Technique | Rationale |
|---|---|---|
| Countdown Timer (60s) | BVA / State Transition | Boundary at 0s (button enables); timer state drives button enabled/disabled |
| Button State Control | State Transition | States: disabled (counting) → enabled (0s) → loading (pressed) → disabled (counting again) |
| Token Invalidation | State Transition | Old token → invalid after new resend; only latest token is valid |
| Rate Limiting (3-5/hour) | BVA / EP | Boundary at the rate limit ceiling; inside limit vs exceeded |
| Token Expiry (15 min) | BVA | Boundary at 15 minutes — token valid just below, expired at/above |
| Loading State / Double-press guard | Specification-based | AC 2.3 mandates loading state to prevent duplicate submissions |
| API Error Feedback | Error Guessing | Server error path: timer must not reset; error toast displayed |
| Success Feedback | Specification-based | AC 4.1 specifies exact success toast text |

## Automation Notes

- **Framework:** WebdriverIO + Appium + Mocha (TypeScript)
- **Spec file (future):** `src/test/mobile/auth/PP-228.resend-verification-email.mobile.ts`
- **Automatable subset (Yes):** TC-001, TC-002, TC-003, TC-004 — timer-driven UI state transitions are fully automatable using `waitUntil` with element attribute checks.
- **Partial automation:** TC-005, TC-006 require email inbox integration; TC-007 requires pre-seeded rate-limit state; TC-008 requires network/server mocking; TC-009 requires time manipulation.
- **Email inbox integration:** Mailosaur API recommended for TC-004, TC-005 to assert email delivery and compare token values.
- **Timer testing approach:** Use `browser.pause()` strategically at BVA boundaries (59s, 60s, 61s); avoid fixed sleeps — prefer `waitUntil(() => element.isEnabled())`.
- **Regression trigger:** Run before every STG release touching email verification, resend cooldown, or rate limit configuration.
