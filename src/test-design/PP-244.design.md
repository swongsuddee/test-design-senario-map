**Platform:** Web (Back-Office / Organizer Portal)
**Framework:** WebdriverIO + Appium + Mocha (TypeScript)
**App:** POPPA Back-Office Web (Organizer Portal)
**Scope:** Organizer initiates account deletion — confirmation dialog, soft-delete, force logout, landing page redirect, and re-login prevention

---

## A. Technique Selection

| Module | Technique | Rationale |
|---|---|---|
| Delete Account button visibility | Specification-based | AC 1.1 requires button to be accessible from Account Settings |
| Confirmation Dialog | Specification-based | AC 1.2 requires dialog before any deletion action |
| Cancel flow (no deletion) | State Transition | Dialog dismiss → account intact — one transition with one clear postcondition |
| Successful deletion | State Transition | Multi-step: confirm → API → logout → redirect |
| Force logout after deletion | State Transition | Session must be cleared immediately after deletion |
| Re-login prevention | Error Guessing | Using deleted credentials must be rejected by the auth system |
| API error on deletion | Error Guessing | Network or server failure during deletion — account must NOT be deleted |

## Automation Notes

- **Framework:** WebdriverIO + Appium + Mocha (TypeScript)
- **Spec file (future):** `src/test/web/organizer/PP-244.delete-organizer-account.web.ts`
- **Automatable subset:** TC-001 through TC-006 — automatable via WebdriverIO web UI + API
- **Manual / Partial:** TC-007 requires network mock (WireMock / mitmproxy)
- **Critical caution:** TC-004 must use a throwaway account provisioned per-run by a before-hook (create account via API → run deletion test). Never delete the shared test organizer account.
- **Session validation (TC-005):** Assert that the auth token is invalidated by calling a protected API endpoint with the old token and expecting 401 Unauthorized.
- **Regression trigger:** Run before every STG release touching authentication, session management, or account settings
