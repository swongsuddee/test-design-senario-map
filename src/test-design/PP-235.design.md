# PP-235 · [BO][Organizer] Event Running Registered — Test Design

> Requirements → [PP-235_Event_Running_Registered.md](../requirements/PP-235_Event_Running_Registered/PP-235_Event_Running_Registered.md)
> Flow Diagram → [PP-235.diagram.md](./PP-235.diagram.md)
> Jira subtask → [PP-235](https://7-solutions.atlassian.net/browse/PP-235)

**Platform:** Web (Back-Office / Organizer Portal)
**Framework:** WebdriverIO + Appium + Mocha (TypeScript)
**App:** POPPA Back-Office Web (Organizer Portal)
**Scope:** Organizer views, searches, filters participant list and drills into participant detail for a running event

---

## A. Technique Selection

| Module | Technique | Rationale |
|---|---|---|
| Participant List Display | Specification-based | AC 1.1 mandates specific status values: registered, confirmed, checked-in, cancelled — verify each |
| Search by Name | EP / Error Guessing | Valid name match, partial match, no-match, empty string — equivalence partitions |
| Filter by Status | Decision Table | 4 status values × present/absent data combinations |
| Participant Detail | Specification-based | AC 2.2 mandates 4 fields: Name, Ticket Type, Payment Status, Check-in Status |
| API Error Handling | Error Guessing | Simulate API failure / empty dataset to verify UI error state |
