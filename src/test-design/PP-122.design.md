**Platform:** Mobile (iOS)
**Framework:** WebdriverIO + Appium + Mocha (TypeScript)
**App:** POPPA Flutter App
**Scope:** Lock Screen push notification card layout, stacking, deep link, and timestamp

---

## A. Technique Selection

| Module | Technique | Rationale |
|---|---|---|
| Card Layout — Image presence | Decision Table | Two conditions: Image present (Yes/No) × Badge count (>0 / 0) → 4 combinations drive layout variants |
| Card Layout — Font Hierarchy | Checklist / Specification-based | AC1 specifies fixed font-weight mapping; verify each element type |
| Notification Stacking | State Transition | States: single card → stacked → expanded. Trigger: 2nd notification from same app |
| Deep Link Routing | State Transition + Decision Table | Route resolved (Yes/No) → 2 outcomes |
| Timestamp Format | BVA | Three regions: < 1h, 1–23h, ≥ 24h — boundaries at 60 min and 24 h |

Notes:
  All test cases are Manual (No automation) or Partial automation because:
  - Lock Screen UI is a system-level iOS surface not accessible via Appium
  - Badge rendering is a native iOS system overlay
  - Notification Service Extension (image download) is outside app process scope
  - Deep link tap on Lock Screen requires device unlock which cannot be automated

  → Automatable scope: POST /push API call to trigger notification payload (API-level)

## Test Data Requirements

| Data | Value / Source |
|------|----------------|
| Test device | Physical iOS device (iPhone) — Lock Screen behavior not reproducible on Simulator |
| STG push trigger | Backend API or Firebase console to send push to specific device token |
| Push payload — with image | `mutable-content: 1` + valid `attachment-url` pointing to a publicly accessible image |
| Push payload — without image | Standard payload; no `attachment-url` field |
| Push payload — badge count | Set `badge` field in APNs payload: `1` for TC-001, `0` for TC-002 |
| Deep link — Event Detail | `poppa://event/{eventId}` — use a valid STG event ID |
| Deep link — Profile | `poppa://profile/{userId}` — use a valid STG user ID |
| Deep link — Home | `poppa://home` |
| Deep link — Unknown | `poppa://unknown/does-not-exist` |
| Timestamp — fresh | Send push and observe within 5 min |
| Timestamp — 1h+ | Send push, wait ≥ 1 hour, or advance device clock |
| Timestamp — 24h+ | Send push previous day, or advance device clock by 24+ hours |

## Automation Notes

- **Framework:** WebdriverIO + Appium + Mocha (TypeScript)
- **Spec file (future):** `src/test/mobile/notifications/PP-122.push-notification-ui.mobile.ts`
- **Automatable subset:** API-level — trigger push via STG backend API, assert push is delivered (via device token registration). UI-level assertions on Lock Screen are **not** automatable with current Appium setup.
- **Manual priority:** All 15 test cases require manual execution on physical iOS device.
- **Regression trigger:** Run before every STG release that touches push notification payload structure or Notification Service Extension.
