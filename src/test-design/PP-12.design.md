# PP-12 · Remote Config — Test Design

> Requirements → [PP-12_Remote_Config.md](../requirements/PP-12_Remote_Config/PP-12_Remote_Config.md)
> Flow Diagram → [PP-12.diagram.md](./PP-12.diagram.md)
> Jira subtask → [PP-12](https://7-solutions.atlassian.net/browse/PP-12)

**Platform:** Mobile (iOS / Android)
**Framework:** WebdriverIO + Appium + Mocha (TypeScript)
**App:** POPPA Flutter App
**Scope:** Firebase Remote Config integration — version checking, force/soft update dialogs, and store redirection

---

## A. Technique Selection

| Module | Technique | Rationale |
|---|---|---|
| Firebase Config Fetch | Specification-based | AC1 specifies exact keys that must be retrievable; verify each key is present and non-empty |
| Version Checking | Decision Table | Two conditions: version comparison (up-to-date / outdated) × isForceUpdate (true / false) → 3 distinct outcomes |
| Update Dialog — Force vs Soft | State Transition | States: no dialog → force dialog / soft dialog → store opened / home; transitions driven by flag values |
| Store Redirection | Decision Table | Platform (iOS / Android) × dialog type (Force / Soft) → correct store URL opened |
| Fetch Failure / Fallback | Error Guessing | Network unavailable → cached/default values used; app must not crash |

## Test Data Requirements

| Data | Value / Source |
|------|----------------|
| Firebase project | STG environment Firebase project |
| versionApp (up-to-date) | Set equal to installed app version (e.g. "1.0.0") |
| versionApp (outdated) | Set to a higher value than installed (e.g. "9.9.9") |
| versionApp (SemVer BVA — minor overflow) | installed = "1.10.0"; set versionApp = "1.9.0" → app must treat installed as up-to-date (no dialog); validates SemVer not lexicographic ordering |
| isForceUpdate = true | Firebase Remote Config key: `isForceUpdate` = `true` |
| isForceUpdate = false | Firebase Remote Config key: `isForceUpdate` = `false` |
| storeUrl | Single Firebase Remote Config key; the app reads this URL and opens it on both platforms via the device OS; value set in STG Firebase console |
| Test device (iOS) | Physical iPhone or iOS Simulator running STG build |
| Test device (Android) | Physical Android device or Android Emulator running STG build |
| Network state (TC-002) | Airplane mode / blocked network for fetch failure scenario |

## Automation Notes

- **Framework:** WebdriverIO + Appium + Mocha (TypeScript)
- **Spec file (future):** `src/test/mobile/remote-config/PP-12.remote-config.mobile.ts`
- **Automatable subset:** TC-001 through TC-006 are fully automatable via Appium UI interaction; TC-007 and TC-008 require partial manual verification for store deep-link destination.
- **Firebase config manipulation:** Use Firebase Admin SDK or Firebase REST API in test setup hooks to set Remote Config values before each test run.
- **Network simulation:** Use Appium's `mobile: setNetworkSpeed` (Android) or Charles Proxy / network conditioner (iOS) for TC-002.
- **Regression trigger:** Run before every STG release that changes app versioning logic or Firebase Remote Config key schema.
