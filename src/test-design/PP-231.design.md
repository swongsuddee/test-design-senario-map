# PP-231 · [BO][Organizer] Organizer Dashboard (Event Detail Dashboard) — Test Design

> Requirements → [PP-231_Organizer_Dashboard.md](../requirements/PP-231_Organizer_Dashboard/PP-231_Organizer_Dashboard.md)
> Flow Diagram → [PP-231.diagram.md](./PP-231.diagram.md)
> Jira subtask → [PP-155](https://7-solutions.atlassian.net/browse/PP-155)

**Platform:** Web (Backoffice — Organizer Portal)
**Framework:** WebdriverIO + Appium + Mocha (TypeScript)
**App:** POPPA Backoffice (Organizer)
**Scope:** Organizer Event Detail Dashboard — metrics display (Pie Chart, Revenue, Tickets Sold, Participants, Ticket Metrics table), date range filtering, data accuracy (confirmed-payment only), and CQRS read model correctness

---

## A. Technique Selection

| Module | Technique | Rationale |
|---|---|---|
| Dashboard Load — All Metrics | Specification-based | AC 1.1–1.3 define exact components that must appear; checklist verification |
| Pie Chart — Ticket Distribution | Decision Table | Conditions: number of ticket types (1, multiple, 0) × tickets sold (yes/no) |
| Summary Metrics Accuracy | Specification-based / EP | Revenue, Sold, Participants must match database of confirmed payments only |
| Ticket Metrics Table | Specification-based | Per-type breakdown — name, quantity sold, revenue per type |
| Date Range Filter | State Transition / EP | States: default → picker open → valid range selected → data updated; invalid range rejected |
| Data Accuracy — Confirmed Only | Decision Table | Payment statuses: confirmed (counted), pending (excluded), failed (excluded) |
| CQRS Read Model | Specification-based | AC 3.2 — metrics served from separate read model; write path changes do not affect read |
| API Error Handling | Error Guessing | Dashboard behaviour when API is unavailable |

## Automation Notes

- **Framework:** WebdriverIO + Appium + Mocha (TypeScript)
- **Spec file (future):** `src/test/mobile/organizer/PP-231.organizer-dashboard.mobile.ts`
- **Automatable subset (Yes):** TC-001, TC-002 (partial — legend/label), TC-003 (with API assertion), TC-004, TC-005, TC-007 — all UI element visibility and data binding checks.
- **API-level assertions:** Use WebdriverIO's built-in `browser.call()` with `axios` to cross-check metric values against the Event Summary Query API response directly (TC-003, TC-004, TC-008).
- **Pie Chart:** Assert via chart legend labels and counts rather than pixel rendering; consider adding `data-testid` attributes to each pie slice for automation.
- **Date Range Filter:** Use `$('[data-testid="date-range-picker"]')` and assert the metrics container re-renders (element text changes) after filter application.
- **Regression trigger:** Run before every STG release touching event summary API, ticket metric consumers, or dashboard UI components.
