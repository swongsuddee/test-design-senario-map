**Platform:** Web (Backoffice — Organizer Portal)
**Framework:** WebdriverIO + Appium + Mocha (TypeScript)

**Scope:** Organizer Event Detail page — event information display (name, date/time, location, images, ticket types) and GET Event API integration correctness

---

## A. Technique Selection

| Module | Technique | Rationale |
|---|---|---|
| Event Detail — Core Fields | Specification-based / Checklist | AC 1.1 defines exact fields: name, date/time, location, images, ticket types; verify each |
| API Integration (GET /event/:id) | Specification-based / State Transition | AC 1.2 — data comes from API; verify binding from 200 response to UI |
| Image Display Variants | Decision Table | Conditions: images present (yes/no) × count (one/many) → gallery vs placeholder |
| Ticket Types Variants | Decision Table | Conditions: ticket types count (0, 1, many) → empty state vs single vs list |
| API Error Handling | Error Guessing / State Transition | 404 (not found) and 500 (server error) cases with distinct error states |
| Data Freshness / Ownership | Specification-based | Organizer sees only their own events; another organizer's event is inaccessible |

## Automation Notes

- **Framework:** WebdriverIO + Appium + Mocha (TypeScript)
- **Spec file (future):** `src/test/mobile/organizer/PP-234.organizer-event-detail.mobile.ts`
- **Automatable subset (Yes):** TC-001 through TC-005 — all element visibility and content assertion on the Event Detail page using standard WebdriverIO selectors.
- **API cross-validation (TC-004):** Use `browser.call(() => axios.get('/api/event/:id'))` to fetch and compare API response against UI-rendered text values.
- **Image assertion:** Check that image `src` attributes are non-empty and the `<img>` elements are visible; use `isDisplayed()` rather than pixel-level checks.
- **Ticket Types assertion:** Assert list item count matches expected ticket type count from API response; check text content of each row.
- **Error states (TC-006, TC-007):** Use WebdriverIO's URL navigation (`browser.url()`) for 404 testing; use `browser.throttle()` or WireMock for 500 simulation.
- **Regression trigger:** Run before every STG deploy touching the Event Detail UI component (PP-157) or the GET Event Detail API endpoint (PP-118).
