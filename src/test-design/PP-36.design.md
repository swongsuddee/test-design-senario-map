Notes:
  - This is a POC (Proof of Concept) story. The scope is exploratory and the test
    cases reflect the DoD rather than a fully specified feature.
  - All test cases are Manual (No) because:
    (a) The chat SDK and UI element structure are not yet stable enough for Appium
        automation.
    (b) TC-001 and TC-006 are process/documentation checks, not UI interactions.

  - Once the POC is accepted and a production implementation is planned, these test cases
    should be converted into proper automated regression tests with stable element IDs.
  - Story status is "Review Code" — test design reflects what can be verified at this stage.

## Automation Notes

- **Framework:** WebdriverIO + Appium + Mocha (TypeScript)
- **Spec file (future):** `src/test/mobile/chat/PP-36.open-chat-poc.mobile.ts`
- **Automation scope:** This is a POC story. Automation is deferred until the chat feature is promoted from POC to a production implementation. Current blockers:
  - Chat UI element accessibility IDs are not defined.
  - The chosen SDK and its Flutter wrapper may not expose standard accessibility attributes.
  - Two-device coordination for message delivery (TC-004) requires a dedicated test framework setup.
- **Post-POC automation:** Once the solution is decided and a production feature is scoped, automate TC-003 (send message) and TC-004 (message delivery) using Appium + WebdriverIO with well-defined element selectors.
- **All 6 test cases are Manual** at this POC stage.
- **Regression trigger:** Not applicable at POC stage. Revisit when feature is promoted to production sprint.
