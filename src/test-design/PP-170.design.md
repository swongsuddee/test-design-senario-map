> Requirements → [PP-170_Document_Review_Approval_Logic.md](../requirements/PP-170_Document_Review_Approval_Logic/PP-170_Document_Review_Approval_Logic.md)
> Flow Diagram → [PP-170.diagram.md](./PP-170.diagram.md)

**Platform:** Web (Back Office — Admin Portal)
**Framework:** WebdriverIO + Appium + Mocha (TypeScript)
**App:** POPPA Back Office (Admin BO)
**Scope:** Admin document review, Approve / Reject Agency with reason validation, and internal audit log

---

## A. Technique Selection

| Module | Technique | Rationale |
|---|---|---|
| Document Display by Type | Decision Table | Two profile types (Corporate / Individual) × correct document set — exhaustive 2-case table |
| Approve Action | State Transition | Status transitions: PENDING_REVIEW → APPROVED; side effects: email + permission unlock |
| Reject — Missing Reason | Specification-based + Error Guessing | AC explicitly blocks reject with no reason; test the blocking guard |
| Reject — With Reason | State Transition | Status transitions: PENDING_REVIEW → REJECTED; reason stored |
| Internal Log | Specification-based | reviewedBy + reviewedAt must be present after any review action |

## Automation Notes

- **Framework:** WebdriverIO + Appium + Mocha (TypeScript)
- **Spec file (future):** `src/test/web/admin/PP-170.document-review-approval.web.ts`
- **State setup:** Use API (PATCH verificationStatus) to place Agency in PENDING_REVIEW before each test
- **State teardown:** Reset Agency status via Admin API after test to allow re-runs
- **Email assertion:** Integrate with Mailtrap API for automated email verification (TC-003 step 5)
- **Log assertion:** Assert via GET /admin/organizers/:id that reviewedBy and reviewedAt are populated
